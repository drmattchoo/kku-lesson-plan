01 / The steps you do

Your side, start to finish

Claude Code writes the code. These are the things it can't do for you — decisions, accounts, materials, and the two checks where a human eye is the whole point.

Before you open Claude Code

Build one clean official KKU lesson-plan template

One tidy .docx in the real KKU แผนการสอน format, plus a written list of every field it must contain. Decide one detail: is the timed table a real Word table, or laid out as text? It changes how the tags are written.

Collect 2–3 real, messy course specs

The actual hectic มคอ-3 files, not tidy examples — plus a sample slide deck. These are what extraction gets built and tested against, so it survives the real mess.

Sort out KKU Google sign-in ask IT first

Confirm with the Office of Digital Technology whether faculty @kku.ac.th accounts sign in via Google or Microsoft. Then create the sign-in app (Google Cloud, or Microsoft if that's the answer) restricted to your faculty domain. This is your own free project — KKU doesn't provide it.

Get the university API key + pick the runtime model

The key the deployed app uses to call GPT-5.5 or Sonnet for extraction and outlines. Separate from your Claude Code subscription.

Decide your Claude plan for the build

Pro has no Opus, so the hard extraction module falls back to Sonnet. Max — or authenticating Claude Code with an API key — unlocks Opus and lets the build run past usage walls.

Drop your prepared files into the repo

Put the template, the sample specs, and your .env alongside the ready-made files (see section 03), then open Claude Code.

During the build — one phase per window

Build module by module, commit when green

Open Claude Code (it reads CLAUDE.md + PROGRESS.md automatically). Tell it: "Do Mx per the contract. Test first. Stop when green." Commit each green module. End every window on a green commit and a PROGRESS.md update — that's what makes a usage reset cost you nothing.

Stop at the template gate human check

After the template-render module, open the produced .docx in Word. Is it a real, acceptable KKU lesson plan? Fix the template and repeat until yes. Build nothing else until this passes.

Stop at the extraction gate human check

Run extraction on your real messy specs. The test isn't "did the AI get it perfect" — it's "can I fix the wrong rows in under a minute?" The correction screen carries the mess.

Let the loop batch the boring middle (optional)

Once past the design-heavy modules, build-loop.sh can run the repetitive ones unattended in a window — point it at an API key if you want it to push past a usage wall.

At the end

Deploy ask IT

A managed host (Render / Railway / Fly.io) is the easy path — or hand it to KKU IT if they want it on university infrastructure, which is likely given the course material and key involved.

02 / Files & materials you prepare

What only you can bring

Three files and a couple of accounts. Everything else is generated by Claude Code or already made for you.

A
templates/kku_lesson_plan.docx

Your clean official KKU template. You provide the base document and field list; Claude Code adds the fill-in tags. This is the canonical mould every lesson plan is poured into.

B
tests/fixtures/ — real course specs + a slide deck

2–3 genuinely messy มคอ-3 files and one sample .pptx. Used to build and prove the extraction against reality, not against a tidy ideal.

C
.env — your secrets (never committed)

University API key + chosen model, and the Google sign-in client ID / secret. Add it to .gitignore so it never lands in git.

D
Two accounts set up (not files)

The Google (or Microsoft) sign-in app restricted to @kku.ac.th, and your Claude plan / API-key decision for the build.

03 / The repo you hand to Claude Code

Everything in place at handoff

Drop the ready-made files (teal) next to the ones you prepared (amber). The moment Claude Code opens this folder it has the full plan, the contracts, and the state tracker.

CLAUDE.md

The brain of the handoff — read free at the start of every session. Holds the schemas, API, build order, and conventions. Keep it tight.

PROGRESS.md

The state tracker. Read it first each window, update it last. This is what carries you across every usage reset.

build-loop.sh

Optional headless builder for the repetitive modules, with the per-module model choices baked in (Haiku / Sonnet / Opus).

docs/architecture.html

The full reference — system diagram, data model, pipeline, template binding. Where Claude Code looks up any detail.
