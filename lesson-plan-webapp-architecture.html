System Architecture · v1

แผนการสอน Generator
web app architecture

An instructor-facing tool that reads a course spec (มคอ-3) and lecture slides, extracts the outcome structure, and drafts a university-format lesson plan one lecture at a time — with a human review gate at every step.

Input มคอ-3 (+ optional slides / brief) Output .docx (KKU template) LLM GPT-5.5 / Sonnet (swappable) Pattern extract-once · generate-in-loop
01 / Overview

What the app does, and the shape of the problem

A course has one outcome structure (PLOs, CLOs, the CLO→PLO map) and many lectures. So the work splits cleanly into two phases. Extract once: parse the unorganised course document and slide deck into a single structured object the instructor can verify. Generate in a loop: for each lecture the instructor picks, draft a timed outline, let them edit it, then render the .docx. The expensive, error-prone parsing happens once; the cheap per-lecture generation repeats.

You already have most of phase one

Your lesson-plan skill already turns PPTX + มคอ into แผนการสอน with CLO mapping. The web app is the interactive shell around that logic: the extraction and template-binding code lifts almost directly out of the skill — the new work is the UI, the two review gates, and the API plumbing.

03 / Tech stack

Recommended stack, and why

The document-processing libraries you need — python-pptx, python-docx, docxtpl — are all Python, and they're the same ones your lesson-plan skill uses. Keeping the backend in Python means zero reimplementation of the hard parts.

Frontend
React + Vite

A 5-step wizard with local form state. Claude Code scaffolds this fast. Plain HTML/JS works too if you want fewer moving parts.

Backend
FastAPI (Python)

Async, typed request/response models via Pydantic — which double as your data schemas. Hosts all doc + LLM logic.

Doc parsing
python-pptx / -docx

Slides and the มคอ-3 to text + structure. Lift straight from the skill.

Templating
docxtpl (Jinja2)

Write Jinja tags directly inside the Word template. Loops and conditionals render the lesson-plan table natively.

LLM
Provider interface

One thin abstraction; OpenAI-format and Anthropic-format implementations behind it. Switch by config.

State
Session JSON

One JSON blob per session on disk or SQLite. No real DB needed for a single-instructor tool.

04 / Data model

The objects that flow through the pipeline

Four schemas carry the whole app. Define them once as Pydantic models; they validate every LLM response and every API payload. This is the contract Claude Code builds against.

Instructor profile — collected in Stage 1
InstructorProfile {
  name, title, department, faculty, university,
  courseCode, courseName, academicYear, semester, section
}
Extracted course — produced once in Stage 2
ExtractedCourse {
  courseCode, courseName,
  PLOs: [{ id, text }],                       // program-level outcomes
  CLOs: [{ id, text, ploRefs: [ploId] }],     // course outcomes, mapped to PLOs
  lectures: [{
    id, week, topic, name,
    durationMin,                              // e.g. 60
    cloRefs: [cloId]
  }]
}
Lecture outline — generated per lecture in the loop (Stage 4)
LectureOutline {
  lectureId, totalDurationMin,
  keyPoints: [{
    seq, title, content, durationMin,         // 5–10 min each
    teachingMethod,                           // "lecture" | "interactive" | "quiz"
    cloRefs: [cloId],
    materials, assessment
  }]
}
Render context — assembled at export
LessonPlanContext = InstructorProfile + selected Lecture
                  + its CLOs/PLOs + edited LectureOutline
                  // → docxtpl renders this into the KKU template
05 / Pipeline

The seven steps, with their inputs and outputs

Each stage has a clean input → output contract. Stages 4–7 form the repeatable loop body.

1
Instructor fills in their details

A simple form. These values become the lesson-plan header and aren't worth asking an LLM to guess.

in · keyboard out · InstructorProfile
2
Upload documents · extract structure

Instructor uploads the course spec (มคอ-3) — the only required file — and optionally a slide deck. Backend parses to text, then the LLM returns the full ExtractedCourse: PLOs, CLOs, the CLO→PLO map, and every lecture with topic, name, duration, and CLO refs. The spec drives this; the structure comes from it.

in · มคอ-3.docx (+ slides.pptx, optional) out · ExtractedCourse (JSON)
3
Instructor selects lecture(s) to plan

The extracted lecture list is shown for review. The instructor confirms it's correct and ticks one or more lectures to generate plans for.

in · ExtractedCourse out · selected lectureId[]
⊳ review gate · instructor verifies extraction before any generation
↻ loop body · runs once per selected lecture
4
LLM suggests a timed outline

For the chosen lecture, the model proposes 4–8 key points for a 60-min slot (5–10 min each, summing to the duration), each with suggested content and a teaching method — lecture, interactive, or quiz — and the CLOs it serves. Grounding is optional and adaptive: slides ground it in real material, a short brief grounds it in the instructor's intent, and with neither it works from the topic + CLOs.

in · lecture + CLO context (+ slides / brief, optional) out · LectureOutline (draft)
5
Instructor edits the outline

Editable table: reorder, retitle, adjust minutes, change teaching method, add or drop key points. A live check confirms the minutes still sum to the lecture duration.

in · draft LectureOutline out · approved LectureOutline
⊳ review gate · nothing exports until the instructor approves
6
Render into the KKU .docx template

Backend assembles LessonPlanContext and feeds it to docxtpl. Jinja tags in the template render the header, the CLO/PLO mapping table, and the timed key-point table. Out comes a finished แผนการสอน.

in · LessonPlanContext out · lesson-plan.docx
7
Deliver — single file or batched ZIP

Download the one lecture, or — because Stages 4–6 are a loop — batch every selected lecture and return a ZIP of plans for the whole course.

in · rendered .docx files out · download / .zip

07 / Model abstraction

One interface, two providers

GPT-5.5 and Claude Sonnet speak different request formats and return structured output differently (OpenAI JSON-schema response format vs. Anthropic tool-use). Hide that behind one interface so the rest of the app never knows or cares which model ran.

class LLMProvider:
    def extract_course(self, mco_text, slides_text) -> ExtractedCourse: ...
    def generate_outline(self, lecture, slides_text, clo_ctx) -> LectureOutline: ...

# two implementations behind the same interface
OpenAIProvider     # GPT-5.5  · response_format = json_schema
AnthropicProvider  # Sonnet   · forced tool-use for structured output

def get_provider(cfg) -> LLMProvider:
    return {"gpt": OpenAIProvider, "claude": AnthropicProvider}[cfg.model](cfg.api_key)

Both methods must return JSON validated against the Pydantic schemas. On a validation failure, retry once with the error appended to the prompt before surfacing it to the instructor.

Prompt sketches

Extraction — "From this มคอ-3 and slide deck, extract PLOs, CLOs, the CLO→PLO mapping, and every lecture (topic, name, duration, CLO refs). Return only JSON matching the ExtractedCourse schema."

Outline — "For lecture «{name}» ({duration} min) produce 4–8 key points of 5–10 min each, summing to {duration}. Per point: title, brief content, a teaching method (lecture / interactive / quiz), and CLO refs. If grounding material (slides or a brief) is provided, base the content on it; otherwise derive it from the lecture topic and CLOs. Return only JSON matching LectureOutline."

08 / Template binding

Where data meets the Word template

With docxtpl you put Jinja2 tags directly into the KKU .docx you already have. No layout code — the template is the layout. The binder just hands it a context dict.

{# header #}
{{ instructor.name }} — {{ course.courseCode }} {{ course.courseName }}
{{ instructor.academicYear }} / ภาคเรียน {{ instructor.semester }}

{# CLO → PLO mapping table #}
{% for clo in lecture.clos %}
  {{ clo.id }} | {{ clo.text }} | {{ clo.ploRefs | join(", ") }}
{% endfor %}

{# timed key-point table — the body of the แผนการสอน #}
{% for kp in outline.keyPoints %}
  {{ kp.seq }} | {{ kp.title }} | {{ kp.durationMin }} นาที | {{ kp.teachingMethod }}
{% endfor %}
Reuse point

If your lesson-plan skill already renders แผนการสอน into a template, this whole section may be a copy-paste. The web app's only change is sourcing the context from the edited LectureOutline instead of generating it inline.

09 / Security & config

Keep the key on the server

  • The university API key lives in a server-side .env — never shipped to the browser, never in client JS, never in a response body.
  • Model choice and key are read once at startup into a config object; the provider factory reads from it. Swapping GPT-5.5 ↔ Sonnet is a one-line config change, no redeploy of the frontend.
  • Uploaded files are processed in a temp dir and discarded after extraction; only the structured ExtractedCourse JSON persists in the session.
  • Rate-limit the /extract and /outline routes — they're the only ones that cost money.
10 / Build sequence

The loop Claude Code can process

Eleven self-contained modules, each with an input → output contract and a one-line test. Build and verify them in order; every module is small enough to hand to Claude Code as a single task. Amber modules are where your existing skill carries most of the weight.

M0Scaffold + confignew
env (API_KEY, MODEL) → config object · provider factory stub Test: server boots, /health returns the active model name.
M1LLM provider interface
prompt → validated JSON · OpenAI + Anthropic impls Test: both providers return a schema-valid object for a trivial prompt; factory switches by config.
M2Document loaderslift from skill
.pptx / .docx → structured text Test: a real มคอ + deck parse to non-empty text with slide/section boundaries intact.
M3Extraction servicelift from skill
doc text → ExtractedCourse Test: output validates; CLO→PLO refs all resolve; lecture count matches the deck.
M4Instructor form (Stage 1)
UI inputs → POST /api/session Test: submitting returns a sessionId; required fields validated client-side.
M5Upload + topic list (Stages 2–3)
file upload → /api/extract → selectable lecture list Test: uploaded files yield a checkbox list of lectures the instructor can confirm.
M6Outline service (Stage 4)partial reuse
lectureId → LectureOutline draft Test: 4–8 key points; minutes sum to lecture duration; each method ∈ {lecture, interactive, quiz}.
M7Outline editor (Stage 5)
draft → editable table → PUT /api/outline Test: add/remove/reorder works; live total updates; warns if minutes ≠ duration.
M8Template binder (Stage 6)lift from skill
LessonPlanContext → .docx Test: rendered file opens in Word; header, CLO/PLO table, and key-point table populate.
M9Batch loop export (Stage 7)
